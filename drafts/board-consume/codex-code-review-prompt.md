## Specifically probe these
- **Scope enforcement coverage.** Verify every endpoint requires the right scope: `/publish` → `publish`; `/inbox` → `read`; `/claim` → `read`; `/ack` → `ack`; `/quarantine` → `ack`. Any endpoint that defaults open? Any way for a `publish`-only token (legacy) to read or ack? Default for legacy tokens (no `scopes` column data) — is `publish`-only the right minimum and not too permissive?
- **`supervisor_decision` forgery guard.** Token check is `tokenRow.agent === 'alex' && hasScope(SUPERVISE)`. Is there any path that bypasses this — `board-publish.js` CLI (which trusts `--from`)? The mesh bridge classifying as `decision` (it doesn't post `supervisor_decision` — but verify). What if someone enrolls a *different* token also named `agent='alex'` with supervise scope — the gate is "I have a token mapping to alex+supervise" which is fine but record the operational impact. Are there UI/text-similarity attacks (`alex` vs `аlex` with a Cyrillic а)?
- **Inbox SQL & client scoping.** `WHERE recipient_agent=? AND acked_at IS NULL AND status != 'dead' AND (claimed_at IS NULL OR lease_until IS NULL OR lease_until < ? OR claimed_by = ?)`. Logic right? Could a malicious-but-authenticated agent see another's inbox via crafted limit/SQL? (Prepared statements, but verify no string interpolation.) For client-bound tokens, is the post-query JS filter the right place, or should it be a `WHERE` clause? Performance with 200-row pages?
- **Claim atomicity & lease expiry.** Read-then-UPDATE inside `withTx` + WAL — is this *actually* atomic against another claim attempt landing in the same instant? `claimed_at` set to `now()`, `lease_until` set to `now()+leaseS` — does the **comparison** `lease_until < ?` (current time) handle clock skew or ISO sorting reliably? Self-renewal updates `claimed_at` to `now()` (which is correct for restarting the lease) — confirmed correct?
- **Ack atomicity & idempotency.** `UPDATE ... WHERE delivery_id=? AND recipient_agent=? AND acked_at IS NULL` followed by a SELECT to disambiguate 0-rows-changed. Is the SELECT-after-UPDATE inside the same `withTx`? (It is.) Is the `first_ack:false` response on same-agent re-ack correct, or does it leak info to a wrong-agent? (Wrong agent gets 403 before the SELECT.)
- **Quarantine race.** Concurrent ack and quarantine on the same delivery — the design says quarantine refuses if `acked_at` is set. What if `INSERT OR REPLACE` into `gateway_quarantine` succeeds but the subsequent `UPDATE deliveries SET status='dead'` is preempted by a successful ack? Should both happen in `withTx` (they do)? Any inconsistency window?
- **Drainer guard semantics.** `UPDATE deliveries SET status='read' WHERE delivery_id=? AND status='pending' AND acked_at IS NULL`. What about `status='dead'`? — guard implicitly skips it (status check), good. What if a delivery goes pending → acked → drainer runs → would skip (correct). What if drainer races: drainer SELECTs pending, gateway acks, drainer UPDATEs — UPDATE skipped (correct). What if drainer running first → status='read' → gateway acks → ack UPDATE has `acked_at IS NULL` guard, succeeds, status becomes 'acked' (correct).
- **HTTP-layer routing.** Regex `^/inbox(?:\?(.+))?$` etc. — correct anchors, no path traversal, query parsing via custom `parseQuery` is whitespace/encoding safe? What does `/inbox/../publish` resolve to? Does the gateway's existing `req.url` normalization matter (it doesn't, Node serves raw — but the regex anchors prevent traversal)? Is `decodeURIComponent` on the delivery_id necessary or attack-amplifying?
- **Backward compat & migration safety.** `ALTER TABLE ... ADD COLUMN` on a WAL DB with the drainer running — is the swallow-on-`duplicate column` idempotent? `NOT NULL DEFAULT 0` on `delivery_attempts` — sqlite handles that ALTER correctly?
- **Action-gate preservation.** The gateway code paths NEVER call agent handlers or external services. Receipt of a fact (even `supervisor_decision`) only inserts/updates DB rows. Confirm: no code path here executes anything beyond DB writes + HTTP response.
- **No fact-type smuggling.** Could a caller publish under a different fact_type than what's in their `can_produce`, by going through a different path (e.g., the supervisor_decision branch)? Trace the order: scope check → forgery guard → publish-scope check → `authzProduce` → registry validate → core write. Any branch that skips `authzProduce`?
