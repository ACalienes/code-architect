## What changed (files)
1. `prototype/shared-layer/board-consume-cfo.js` (NEW) — CFO's consumer: a `supervisor_decision` handler on the Phase-2 `board-consume-lib`. On `approve` of CFO's **own** draft → writes a local `<draft>.ready.json` provenance sidecar **and** echoes a `status_update{status:'ready_to_send'}` back to the Board. Never sends money.
2. `prototype/shared-layer/board-consume-cfo.test.js` (NEW) — 11 tests (own-draft approve, other-agent noop, reject/dismiss noop, missing/again-missing ref → quarantine, **path traversal refused**, non-alex source refused, echo transient vs permanent, idempotent re-approval).
3. `prototype/shared-layer/board-supervisor.js` — `/action` approve/reject/dismiss now publishes **`supervisor_decision`** (was plain `decision`) with a generated `supervisor_action_id` (UUID) + trusted `context{subject_source_agent, draft_ref}` looked up from the decided fact; comment still publishes `status_update`. The "already decided" dedup query now unions `('decision','supervisor_decision')`. Distinct `alex:sd:<action>:<id>` idempotency prefix.
4. `prototype/shared-layer/registry.js` — `supervisor_decision` gains optional `context` (object: `subject_source_agent`, `draft_ref`; `additionalProperties:false`); `status_update` gains optional `draft_ref`.
5. `prototype/shared-layer/board-emit-cfo.js` — the `cfo-draft` fact now carries `payload.draft_ref` (the draft's CFO_DIR-relative path). **Idempotency key unchanged** (`cfo:draft:<relpath>`).
6. `prototype/shared-layer/gateway-enroll.js` — adds `--scopes=publish,read,ack` (order-independent flag; default stays `[publish]`).

## Please specifically scrutinize
1. **No auto-send is structural, not a flag (§12.7).** Confirm there is genuinely no code path from a delivered `supervisor_decision` to a money-movement/invoice-send. The handler's only outward effects are a local sidecar write + a `status_update` publish. Is that the whole surface? Any import or ctx that could send?
2. **Self-filter soundness (§12.1).** Every `supervisor_decision` subscriber receives ALL of them. The CFO handler acts only if `context.subject_source_agent === 'cfo'`. Can a `supervisor_decision` about ANOTHER agent's subject ever cause CFO to write a ready marker or echo? Is `context` the only authorization signal, and is it trustworthy (it's set server-side by the supervisor, which can only publish `supervisor_decision` via the alex+supervise gate)? Is the extra `source_agent==='alex'` re-check meaningful or dead?
3. **Trusted-context vs. spoofing.** The handler trusts `payload.context` rather than re-fetching the subject fact. Is that safe given the Phase-2 forgery gate (`AUTH_GRADE_TYPES` — only alex+supervise publishes `supervisor_decision`)? Is there ANY non-gated path that can persist a `supervisor_decision` with attacker-chosen `context.subject_source_agent='cfo'` + `context.draft_ref`? (Re-check `board-publish.js`, raw `writeFact`, `writeFactValidated`, mesh bridge — the same surfaces Phase-2 closed.)
4. **Path traversal.** `resolveDraft()` confines `draft_ref` to `<cfoDir>/logs/drafts/*.json` (and rejects `.ready.json`). Any bypass — symlink, `..` after resolve, absolute path, null byte, case/Unicode? Could a crafted `draft_ref` write a `.ready.json` outside the drafts dir or overwrite a non-draft file?
5. **Idempotency-key migration hazard.** The supervisor now uses `alex:sd:<action>:<id>`; the gateway dedupes on `(agent,key)` with a request-hash check (a reused key + different body → 409). Confirm: (a) a NEW approval is never shadowed by a legacy `alex:<action>:<id>` `decision` fact; (b) re-clicking approve twice dedupes cleanly (same key, same body → 200 idempotent); (c) the CFO echo's `cfo:ready:<draft_ref>` can't collide destructively with the `cfo:draft:<relpath>` emit key.
6. **Dedup-union regression.** With approvals now `supervisor_decision`, does the unioned `decided` set correctly drop approved items from "what needs you" for BOTH heuristic (fact_id) and ca-mesh (message_id) items? Do legacy `decision` facts still dedup?
7. **At-least-once correctness.** On echo transient failure the handler returns `{ok:false}` (un-acked → retried), having already written the sidecar. Both the sidecar write and the echo are idempotent. Confirm a retry storm can't double-surface or diverge. On permanent echo failure (`4xx`) it returns `{ok:false,permanent:true}` → quarantine — is quarantining (vs ack) the right call when the local READY is already written?
8. **Producer payload change.** `cfo-draft` now includes `draft_ref`; `status_update` schema allowlists it. Confirm already-posted drafts are NOT re-posted (idempotency key unchanged) and that adding `draft_ref` can't break the existing CFO emit baseline/seen-state.
9. **Scope/enroll.** `--scopes` parsing (order-independent, default `[publish]`); unknown scope throws (fail-loud). Re-enrolling `alex`/`cfo` creates a NEW active token row — does a stale prior row (e.g. alex's old publish-only token) create any privilege ambiguity? (Deploy plan deactivates priors.)
