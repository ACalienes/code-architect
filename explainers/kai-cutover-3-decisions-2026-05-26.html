01 Private-key custody

D1

Where each agent's private key lives

Each agent gets an Ed25519 keypair; the layer stores only public keys. Each agent's private key must live where only that agent's process can read it — never in kameha-mesh.db or any log. enroll.js generates the keypairs and, as a reference, writes private keys as 0600 files; production custody is Kai's call.
CA default recA per-agent 0600 key file under each agent's home, owned by that agent's user, with the path in its env.
Need from Kai: the custody mechanism + how each agent process reads its own key. (Options: pm2-injected env var · per-agent 0600 file · macOS Keychain · secrets manager.)

02 Database location + owner

D2

kameha-mesh.db path + trusted writer

A dedicated SQLite DB — not extending conductor.db (blast radius + isolation). The dir (0700), the db, and its -wal/-shm sidecars (0600) must be denied to agent users — ownership alone isn't enough; the process boundary depends on it.
CA default rec~/.kameha/kameha-mesh.db, owned by the same trusted user that runs the router/projector service.
Need from Kai: the absolute path, and which unix user owns/runs the trusted writer (the router + projector).

03 Per-client unix isolation — the load-bearing one

D3

Do client repos run as distinct unix users?

● isolation-critical
Physical per-client isolation is OS-enforced: each client's projection lives at <projections>/<client-repo>/ owned projector:<client-group>, dir 2750 (setgid, group r-x, no group write), file 0640 — so only that client's group can read its inbox, the projector writes, everyone else is denied. For that to be real, each client repo must run as its own unix user in its own group (e.g. dag-repo as dagdc, tdb-repo as tdb).
CA default recIf agents currently all run as kai under pm2: ship the pilot on logical scoping + signed integrity now, and treat per-client unix users as the separate "multi-user Mini" project (the install is chown-ready). The logical + identity wall works regardless; this just adds the OS layer when it lands.
Need from Kai: can client-repo processes run as distinct unix users/groups today? If not, confirm the logical-only interim — this is the call that decides whether the physical wall is real on day one or deferred.
