Recommendation

Adopt taylorwilsdon/google_workspace_mcp — the consensus most-maintained, most-complete Google Workspace MCP server. It covers the exact Gmail + Calendar surface the fleet uses today, runs over stdio (drop-in for the launcher pattern), and is MIT-licensed.

But it's a migration, not a swap — tool names change fleet-wide and auth needs a one-time re-setup. No fire: the vendored fix holds, so this happens when you choose, as a scoped project.

03 The honest tradeoff — why this is a project, not a one-liner

Cost 1 · fleet-wide

Tool names change everywhere

Agents call the old namespace and names. The new server uses a different namespace and different tool names — so every agent prompt/script that references email tools needs a coordinated update. Multi-repo, DA-gated.

mcp__gsuite__search_emails
→ mcp__google_workspace__search_gmail_messages

Cost 2 · one-time

Auth re-setup

Your existing OAuth client (id/secret) is reusable, but the new server keeps its own token cache. So it needs one browser consent on the Mini — or a service account, since kamehamedia.com is likely a Workspace domain. Headless thereafter.

existing gsuite-launcher.sh credential
hardening would be re-applied, not lost
