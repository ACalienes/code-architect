01 What's real today vs. what's soft on a single-user box

The OS can't enforce a boundary between processes that are the same user. So the app-level guarantees hold; the OS-level ones are deferred — including, honestly, key-secrecy.

MechanismOn the single-user MiniWhy

02 So name the pilot's boundary precisely

The pilot's real isolation
Logical scoping + signed integrity, among trusted first-party agents. Not OS-enforced against a compromised co-resident process. The hard wall (physical + unforgeable identity) arrives with the user-split — not before.

This is an acceptable posture for a pilot of your own agents — the threat it doesn't yet stop (a malicious agent reading a peer's key or DB) isn't the pilot's threat. But it must be stated this bluntly, so nobody mistakes the soft boundary for the hard one.

03 The decision — ship now, and simplify

Ships in the pilot (real now)

  • Type a fact once → the right agents get it (logical scoping)
  • Clients drain their own scoped deliveries from central
  • Signed facts — integrity, attribution, authz client-binding
  • Schema-at-the-door, transactions, dead-letter, observability
  • Keys as 0600 files + DB 0700/0600 — chown-ready targets

Deferred → the "multi-user Mini" project

  • Per-agent / per-client unix users + groups
  • Physical per-client projection files (the round 4–5 work — built, dormant)
  • Unforgeable identity (keys readable only by their own agent)
  • OS-enforced "denied to agent users" on keys + central DB
  • re-chown every repo/DB/log · per-user pm2 · key redistribution

The physical layer that generated the hardest review rounds is built and chown-ready — it switches on as step one of the multi-user project, zero redesign. It shouldn't ride behind a pilot (real downtime risk on a box that just had a 24h OAuth outage).
