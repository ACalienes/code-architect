02 ≤60s comms — the key requirement

✕ Corrected: ≤60s is already met

The 300–600s in /health is the heartbeat cadence, NOT message-poll. In code, CFO / Conductor / Nami all set MESH_POLL_INTERVAL = 30s — every agent already picks up messages within ~30s. The work order (and "polling won't cut it") misread the heartbeat number as latency.

Dropped

Poll-tighten — unnecessary

Agents are already at 30s for messages. Tightening would fix a non-problem. Removed from Phase 0.

Optional · later · DA-gated

Push-nudge — nice-to-have

Only helps multi-hop chains (A→B→C ≈ 90s at 30s/hop) + snappiness. Build only if multi-hop latency actually bites. The real comms pain is reliability, not speed.
