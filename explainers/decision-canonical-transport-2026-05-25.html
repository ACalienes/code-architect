Code Architect · decision you need to make

Build on the old system, or replace it?

Before we build the shared "brain," one call shapes everything after it: does the new Kameha layer sit on top of the message system we already have — or do we stand up the new one and retire the old?

⏱ The 30-second version

We already have a basic "postal service" between agents (called mesh-api) — it works but was never fully switched on. The new plan adds smart, durable shared memory on top. The question: do we keep the old postal service and build the new features onto it, or build one clean new system and shut the old one down? The danger we must avoid is the third path — running both forever, which is exactly the tangle we're in today.

01 What's on the table

mesh-api the old postal service
  • Already exists, running on the Mini
  • Knows who's allowed to message whom + keeps an audit trail
  • But: under-operated, a buggy reply path, and some agents (like CA) were never even added
Kameha Shared Layer the new brain
  • Durable, typed shared memory + private per-agent inboxes
  • The strict client-data isolation Codex made us redesign
  • Needs a home — and that's the decision

02 Your two real choices

Option A

Upgrade in place

Keep mesh-api; build the new features onto it.
🏚️ Like renovating the existing house — you keep living in it while you add rooms.
Faster to start
  • Reuse the routing + audit that already work
  • Less to build up front; more incremental
But
  • You're building on a foundation with known cracks
  • The strict isolation model doesn't fit cleanly onto the old message design — risk of bolting it on
  • Easy to end up quietly maintaining two overlapping systems
CA recommends
Option B

Build new, retire old

Kameha becomes the one system; mesh-api is phased out.
🏗️ Like building the new house next door, moving in room by room, then selling the old one.
One clean system at the end
  • Isolation + private inboxes are native, not bolted on — the thing that nearly bit us is built in
  • One source of truth — no "which channel?" confusion ever again
  • We're already rebuilding the good parts (routing/audit) anyway
Costs
  • More work up front
  • Have to migrate every agent over carefully
The path we must NOT take: keep both running indefinitely and let them drift. That's how we got the current mess — multiple half-wired channels, messages rotting, CA with no route at all. Whatever we pick, we commit to one canonical system.

03 Why this is the call that matters

◆One system or two
Decides whether we maintain a single clean pipe or babysit two overlapping ones forever.
◆Migration risk
How much careful moving-of-agents we take on — and when.
◆Is isolation native?
The client-data boundary Codex flagged is either built into the foundation, or awkwardly stapled to the old one.

04 My recommendation

✓ Option B — but phased gently, not a big-bang switch

Make Kameha the one canonical system, and retire mesh-api gradually — move agents over one at a time, with the old and new running side by side only during each handoff, then formally shut the old one down on a set date.

Why: the strict client-isolation that Codex made us redesign is natural in the new model and clumsy to graft onto the old one. One source of truth is the entire point of this project. And since we're already re-creating mesh-api's good parts (who-can-talk-to-whom + the audit trail) in the new store regardless, "upgrade in place" saves less than it looks.

The gentle part matters: no risky overnight cutover. Each agent moves when its new path is proven, so nothing drops mid-flight.

One note: the messaging plumbing is Kai's domain, so this is a call we make with Kai at the start of Phase 0 — I bring the recommendation, Kai confirms the infra reality, you make the final call. Nothing gets built until that's settled.
