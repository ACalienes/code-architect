8 · Your two concerns — mapped to architecture

latency layer vs knowledge layer · solve them separately
Concern 1 · Transport layer

Semi-instant communication

"When I'm in the chair giving an agent a task, I don't want to wait 10 minutes for a reply."

What's real today

The "10 minutes" feeling comes from 5 specific agents: ACD, NAMI, Framer, Chronicle, KMG — all set to 600s poll. The rest are at 1-5 min. Mesh average is 40s.

Three plausible fixes

A
Drop slow agents to 60s
Per-agent cadence change or "Alex active" flag
Small
Set poll_interval_seconds: 60 for ACD/NAMI/Framer in mesh-api's agents table (or each agent's local config). Can be conditional: a flag at mesh-api like ?session_active=true tells pollers to drop to fast cadence until you toggle it off.
Pros
Smallest fix. Reuses everything. Reversible. Ships this week.
Cons
Higher steady-state CPU on Mini. Each agent's poller code may need a tweak.
B
Push model — webhooks from mesh-api
mesh-api POSTs to each agent's HTTP port on arrival
Med
Each agent already has its own HTTP port (ACD :3342, conductor :3344, etc.). mesh-api adds a webhook dispatcher: when a message lands for agent X, hit POST X:port/mesh/notify. Agent drains immediately.
Pros
True sub-second delivery. No polling waste. Right answer for next quarter.
Cons
Every agent needs a webhook handler. Failure modes (agent down, retries) need design. NAMI on Render needs separate path.
C
Long-lived connections (WebSocket/SSE)
Agents hold open connection to mesh-api
Large
Each agent opens a persistent connection to mesh-api at startup. Messages stream down instantly.
Pros
Architecturally correct for a real-time mesh. No polling at all.
Cons
Significant rewrite per agent. Connection management, reconnect logic, keepalives. Overkill for current scale.
CA's recommendation

Ship (A) this week for the felt pain. Design (B) as the right next step. (C) only if the mesh grows much larger.

113 knowledge files across 8 silos. Plus 7 file-drops in ~/.kameha/shared/. Plus conductor.db (3.6MB active). Three partial shared surfaces, no unified brain.

Three plausible architectures

A
Extend ~/.kameha/shared/ with structured contracts

Builds on the pattern already working. Cheap. Git-backed (auto-backup cron already commits hourly).
Cons
Filesystem isn't a real database. Concurrent writes risk. Eventually-consistent only.
B
Promote conductor.db to general knowledge store
Add new tables beyond projects/tasks
Med
Conductor is the active shared SQLite (3.6 MB WAL). ACD already reads it directly via env var. Add new tables for any typed cross-agent fact: client_facts, decision_log, crew_assignments (already proposed). Conductor becomes the knowledge orchestrator, not just project orchestrator.
Pros
Real DB (queryable, indexable, transactional). Existing pattern. crew_manifest proposal pilots this exact path.
Cons
Conductor's role expands. Schema migrations need discipline. Concurrent writes need transaction wrapping.
C
New mesh-knowledge HTTP service on Mini
Dedicated service, e.g., port 3343
Large
New service that holds all shared knowledge. Agents query via HTTP. Schema enforced server-side. Could include vector search for semantic recall.
Pros
Clean architecture. Independent of conductor. Room for semantic search / RAG.

Whole new service to maintain. Every agent gets a new client lib. Overkill for current need.

(B) is interesting because crew_manifest is already piloting it. If that lands well, expand conductor's role with 2-3 more typed tables. (A) is a stopgap. (C) only if (B) hits a wall.

These two are architecturally separate. A faster transport without a shared knowledge layer just means agents say "I don't know that" faster. A shared knowledge layer without faster transport means the answer exists but takes 10 minutes to retrieve. You need both — but you don't need them in the same change.

What drives token spend (in order)

1. How much context you stuff into each Claude call. This is the dominant lever. Naive "load entire brain into prompt" vs smart "query first, then compose" = ~10× difference on the same functional capability.

2. Per-agent call volume. Kai logs 3 calls/day average. ACD, Framer, etc. likely much more — we don't know yet.

3. Cache hit ratio. Anthropic's prompt cache discount is 90% on cached reads (Sonnet: $0.30/M vs $3/M). Kai's current 22.6% hit ratio is leaving money on the table — most static prompts could be cached but aren't marked.

4. Model choice. Sonnet is ~4× the price of Haiku. Routing simple tasks to Haiku is real savings. Kai already uses both via model-router.js; question is whether other agents do.

10 · Effective universal brain — the 3-layer model

how each agent pulls just what it needs · cheap + fast first

The point of layering isn't elegance — it's doing the expensive thing last. SQL is free and instant. File reads are free and instant. Vector search is nearly free and 50ms. Claude calls cost money and take seconds. Each agent's pipeline pulls from the cheap fast layers FIRST, then composes a minimal Claude prompt with only what matters.

Layer 1 · Structured

SQLite tables

Schema'd facts: projects, shoots, decisions, crew, deliverables, clients.

Lives in: ~/.kameha/conductor.db
Latency: <1ms
Token cost: $0
Access: direct SQL

Use for: "what's the status of project X", "who's on the Baptist shoot"

Layer 2 · Semi-structured

JSON snapshots

Daily-refreshed reports: briefings, financial snapshots, status digests.

Lives in: ~/.kameha/shared/
Latency: <1ms
Token cost: $0
Access: file read + schema validate

Use for: "today's morning briefing", "current financial context"

Layer 3 · Unstructured

Vector-indexed narrative

Client backstory, voice notes, past decisions, learnings — facts without schema.

Lives in: SQLite-vss or pgvector
Latency: 10-50ms
Token cost: ~$0.001/query
Access: embed task → top-K chunks

Use for: "what did Maria say about her story", "our voice for Direct Builders"

The pipeline per agent task

Key principle: the Claude call (step 4) is the only step that costs money or time. Layers 1-3 (step 2) are nearly free and instant. Composing well (step 3) means Claude gets a tight, relevant prompt.
Caching baked in: the agent's system prompt + any static schema descriptions in step 3 get marked cache_control. Anthropic charges 10% of input rate for cached reads (5-min TTL). For active sessions, 20 calls in 10 min pay full rate once and cached 19 times.

How we ensure each agent pulls the right slice

Discoverability

Universal brain ships with a schema manifest at ~/.kameha/shared/knowledge-manifest.json that tells agents: "for queries about clients use L1 table X; for voice/tone use L3 with prefix Y." Self-describing brain.

Relevance

Vector search returns top-K chunks ranked by cosine similarity to the task embedding. K=5 (≈2,500 tokens) is the sweet spot — enough recall, no bloat.

Freshness

Writes to L1 are immediate (live SQLite). L2 snapshots regenerate daily (existing pattern). L3 re-embeds on write. No agent sees stale data older than 24h.

Performance budget

Each agent's context-composer caps total retrieved tokens at 5,000. If queries return more, it summarizes via cheap Haiku first. Hard ceiling, no surprises.

Per-agent budgets

Each agent gets a daily token quota (configurable). Logged to mesh.db. Alert if any agent breaches threshold — cost surprise becomes impossible.

Fallback

If vector search is down or schema manifest is unreachable, agent falls back to current behavior (its own knowledge dir). Brain is advisory, never load-bearing for liveness.
