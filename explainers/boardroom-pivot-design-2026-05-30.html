3 Project logs ↔ boardroom — the path + first schemas

📌 Correction to the brief (verified against live Mini code): objective, question, and task fact-types are NOT missing — they're already defined in registry.js + FACT_TYPES (added 5/27–5/28), alongside supervisor_decision. So the two schemas the project-logs work needs are new ones, not those. (Kai's task-list #8 is stale on this point.)

Recommendation: Path C — Hybrid

Matches your read, and it's the only one that's both safe and pragmatic given the scale: ~/kai/logs/ is ~220 MB across 150+ heterogeneous files. Backfilling all of it as facts (Path A) is a heavy, schema-exploding, risky migration; pointer-only (Path B) leaves two systems of record. Path C: new events emit as facts going forward · historical files stay as cold reference · every fact carries a source_ref pointer so the boardroom indexes file paths for search. One forward source of truth; history stays where it is until/if it's worth backfilling.

A
Ingest-all-as-facts — single source, fully queryable, but heaviest migration + schema design for 150+ files. Overkill now.
B
Pointer-only — lightest, but two systems of record (the risk you flagged). Boardroom never holds the real data.
C ✓
Hybrid — forward facts + source_ref index to cold files. Phased, pragmatic, one forward SoR.

The two fact-types to add first (serve project logs directly)

Modeled on the real shapes of projects.json (project records) and production-log.jsonl (shoot events). Written in the existing registry.js S(required, props) style; source_ref is the Path-C pointer.

// projects.json → a queryable project record (the boardroom's project model)
project: { current: '1', versions: { '1':
  S(['project_id', 'name'], {
    project_id: { type: 'string' },
    name:       { type: 'string' },
    client:     { type: 'string' },
    partner:    { type: 'string' },
    workflow:   { type: 'string' },                            // production | content | …
    stage:      { type: 'string' },                            // label or number-as-string
    status:     { type: 'string', enum: ['active','blocked','done','archived'] },
    next_steps: { type: 'string' },
    source_ref: { type: 'string' } } ) } },                      // logs/projects.json#<project_id>

// production-log.jsonl → one shoot/production lifecycle event, linked to a project
production_event: { current: '1', versions: { '1':
  S(['event', 'occurred_on'], {
    event:        { type: 'string' },                           // "PALMETTO BAY — Day 2"
    occurred_on:  { type: 'string' },                           // ISO date
    project_ref:  { type: 'string' },                           // → a project fact's project_id
    client:       { type: 'string' },
    location:     { type: 'string' },
    hours:        { type: 'string' },
    engagement:   { type: 'string' },                           // White-Label | Direct | …
    deliverables: { type: 'string' },
    status:       { type: 'string', enum: ['scheduled','in_progress','completed','logged'] },
    source_ref:   { type: 'string' } } ) } },                     // logs/production-log.jsonl#<eventId>

These two unlock the project board: project gives the queryable record (stage, client, next steps), production_event the lifecycle stream against it. They also exercise the Path-C source_ref pointer, validating the hybrid before we scale to other log types.

4 board-emit-cfo restart loop — diagnosed

🔍 My first hypothesis (missing --watch) was wrong — and I'm flagging that rather than burying it. The script is launched with --watch and is currently stable at 22h uptime. The real cause is different.
  • Root cause: two things named "watch" collided. PM2's own process flag watch: true is set (in dump.pm2) with no ignore_watch, so PM2 reloads the process on any file change in cwd = /Users/kai/shared-layer — and sync-repos.sh ff-pulls + redeploys touch that dir. The "13 restarts in 16 min" was a clustered redeploy/sync burst, not a clean-exit loop. The benign SQLite warning was a red herring.
  • Fix (proposed, not applied): turn OFF PM2's filesystem watch, keep the script's own loop flag, persist it: 
    pm2 delete board-emit-cfo
# relaunch with PM2 watch OFF, script --watch ON
CFO_DIR=/Users/kai/CFO pm2 start board-emit-cfo.js --name board-emit-cfo \
  --no-watch -- --watch
pm2 save
    …and codify it in the new shared-layer/ecosystem.config.js (watch:false, autorestart:true, args:'--watch', env:{CFO_DIR:'/Users/kai/CFO'}) so it survives pm2 resurrect.
  • Secondary (real, separate) issue: CFO_DIR=/Users/kai/CFO is set, but ~/CFO/logs/drafts does not exist on the Mini — so the emitter surfaces financial alerts but emits zero draft/close facts. This is the same path Phase-3's CFO consumer depends on. Needs: confirm where CFO actually writes drafts on the Mini, then fix the sub-path. (Note: this is also why a Mini-side Phase-3 CFO consumer would see nothing — reinforcing the laptop-hosted consumer decision.)
  • Verify after fix: pm2 describe shows watch ✘ / args --watch ✔; restart count holds steady across a sync-pull; uptime climbs; the [emit-cfo] watching … line prints once.
