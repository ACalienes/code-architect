Making the board confirm, not just show.

You asked the right question: where's the acknowledgement that a message was received and absorbed? Today the board shows things; it doesn't confirm them. This is the plan to add the two missing layers — acknowledgement (got it, logged it) and action (did the right thing, still gated).

Four principles (non-negotiable)

One writer

Agents read their own inbox files; acks flow back through a single writer. No N-agents-writing (that's what caused "database locked" before).

Proof from the agent

The acknowledgement comes from the agent's own logic after it logs — not auto-stamped by the mailman, or it's the same hollow "read" we have now.

Seeing ≠ doing

Awareness facts just inform. Work & tasks always route through your approval gate — never auto-chain.

Re-auditable

Every ack & action is a row you can re-check. The ack is the visible signal; the audit is the truth.

Phase 1 — Acknowledgement (do first, lower risk)

When an agent consumes a fact, it logs it and acks. The ledger shows: delivered → read → ✓ acknowledged by Kai (logged).
  1. Add the acked step to delivery (reuse the board's existing ack mechanism).
  2. Build a reusable consumer: reads an agent's inbox → logs to its memory → acks.
  3. Wire Kai first end-to-end (the live CA↔Kai loop) and prove the round-trip.
  4. Upgrade the ledger to show acknowledgements (who logged what).
  5. CA (single-shot) consumes + acks its inbox at session start.

Phase 2 — Action (after ack is proven)

Agents take the right action per type — gated.
  1. Awareness (objective, decision, status): update context, ack, no action.
  2. Question: if answerable, draft an answer (gated) → on approval, post the resolution.
  3. Work order / task: claim → surface for your approval → act; track status back to the board.
  4. Route every action through each agent's existing approval gate. No auto-chaining. Roll out agent-by-agent (Kai first).
