01 What got built today

Three increments, each built the same disciplined way: written as tested reference code, run against an adversarial test, reviewed by the internal Devil's Advocate, and committed locally (nothing pushed to live).

Increment #1

The drainer runner

The always-on ~60s loop every agent rides to check its inbox. At-least-once: if handling crashes, the message comes back next round instead of vanishing. A message that keeps failing gets parked so it can't jam the queue.

Increment #2

Backfill as claims

Pours our accumulated history into the system as quarantined claims, never facts — scrubbed of secrets/PII at the door, invisible to every agent, and only turned into a real, routed fact by a human-gated promotion.

Increment #3

Physical client wall

Each client repo gets its own permissioned file with only its own data in it. Even raw file access can't reach another client — their bytes simply aren't there. A second guard refuses cross-client copies if anything upstream slips.

03 What I'm not claiming (the honest limits)

Each increment's Devil's Advocate pass found real holes. None block a tested reference, but they're written down — not hidden — and tracked for the hardening before this touches live data.

  • The runner is at-least-once, not exactly-once — handlers must be safe to run twice; one-runner-per-agent and crash-proof retry counts are the deploy hardening.
  • Secret scrubbing is best-effort — it catches known shapes (API keys, PEM/JWT/connection strings, emails); unusual or free-text secrets can slip. The real safety net is that claims are quarantined and a human reviews each before it ever routes.
  • The physical wall is proven by content + permissions, not by cross-user denial yet — that last step needs each client file owned by its own OS user on the Mac Mini. That's the deployment test, documented, not faked.
  • Codex review is intentionally deferred to just before deployment — three scoped prompts are drafted and ready, to run against the real ported code, not throwaway prototype specifics.

04 Latency without the cost — why ~60s, and the wake signal

A fair question: why a 60-second loop, and why not faster? The honest answer is that the interval only governs the idle wait. When messages are actually waiting, the loop already drains them back-to-back immediately — so 60s is the worst case for the first message after a quiet spell, nothing more. It also matches the cadence the fleet already runs at.

Why NOT just poll faster

  • ✕ Mostly empty checks — at 1s, ~16 agents do ~1,000 useless reads/min
  • ✕ Contention on one shared database file on one Mac Mini
  • ✕ Rewrites every client's private file far more often (disk churn)
  • ✕ Steals cycles from the 16 other processes on the Mini — for no real gain

The right lever — wake on delivery

  • The sender pokes just the agent that got a message
  • That agent drains immediately — measured ~20ms, not up to 60s
  • Zero extra empty checks; the 60s loop stays as a safe fallback
  • Proven end-to-end: a drainer set to a 10-minute idle still delivered in 21ms on signal
