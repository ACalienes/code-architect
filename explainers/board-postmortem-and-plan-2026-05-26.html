The Board: learn from the dashboard, don't rebuild it.

You said the kai-dashboard was a good effort that's not used at all. Before The Board becomes the same thing with a new name, here's why the dashboard went unused, the rules that follow, and what's worth carrying over.

01 Why the dashboard sits unused

It's a place you have to GO to

It's a web app on localhost:3000, behind a login token, served from the Mac Mini. To see it you navigate to a URL over Tailscale and authenticate. Nothing brings you there — and you live in terminals and on your phone. A board you must remember to open loses every time.

evidence: dashboard.js — app.listen(3000) · authMiddleware · static web build
It duplicates channels you already use

It's a second way to chat with Kai + run tools — but you already do that in the terminal and on Telegram/iMessage. A redundant front door has no reason to be opened.

evidence: dashboard.js wires the same Kai chat + kai-tools as your other channels
No single source of truth underneath it

It rendered whatever it could reach, with no shared, trusted state behind it — so it couldn't reliably answer "what's actually happening across everything." A board you can't trust, you stop checking. (Your own read — confirmed.)

the gap The Board itself fills
It shows status, not "what needs you"

It displays; it doesn't pull you back. Nothing says "Alex, these 3 things need a decision." So there's no hook to return.

02 The rules The Board lives by (so it actually gets used)

Come to you — don't make you come to it

The Board's important posts get pushed where you already are (terminal, the agents you talk to, a notification). The full board is there to pull up, but you never have to.

fixes cause #1
Be the source of truth, not another view

Every agent reads and writes The Board, so it's load-bearing, not optional. It isn't a duplicate window — it's the one place state lives.

fixes cause #2 & #3
Lead with "what needs you"

The first thing it surfaces is decisions/approvals/flags waiting on you — not vanity status.

fixes cause #4
Only ever show live, trusted state

It reflects what the synced system actually knows (the work we just stood up), or it doesn't show it. Trust is the whole game.

fixes cause #3

Point The Board's view AT Conductorrecommended

The project-board view reads Conductor's live data directly (it stays the project store; it just "folds in" as you decided). New cross-agent updates flow as fresh posts on top. No risky copy of 279 rows; nothing lost.

Physically copy active projects/tasks in as facts — true consolidation, one store. More work + a fact-type for projects/tasks, and we'd filter to what's still active first.
