crew_manifest

A new typed record that says who is working what shoot, in what role, confirmed or unfilled. ACD produces it. Conductor stores it. Kai's morning briefing reads it and warns you at T-7, T-3, and T-24h if any role is still open. This is the proposal that turns the Baptist 5/21 catch into a structural fix — and sets the contract pattern for every typed deliverable the mesh will ever produce.

Why this exists

the Baptist 5/21 framing
Current state

Crew info is prose. No machine-readable view of "who's on what shoot."

ACD produces a production_strategy document that mentions who should shoot what — embedded in narrative context, not exposed as structured data. Kai has no way to query "is the Baptist shoot fully staffed?" without reading text.

Sebas's AM-only constraint was logged in scoping. The PM 2nd-camera slot was never converted into action. Alex personally caught the unstaffed slot at T-2 days. The system tolerated a passive "logged but unactioned" state from confirmation through almost shoot day.

After this proposal

Every shoot has a typed crew_manifest. Unfilled roles surface in the morning brief at T-7, escalate at T-3, red-alert at T-24h.

ACD composes a crew_manifest as a structured sibling to the strategy doc. Conductor stores it as queryable rows. Kai's existing 5:30 AM briefing runs one SQL query against it and surfaces unfilled roles. T-3 also fires a Telegram push outside the email window.

The Baptist incident class becomes observable. Confirmed shoot with no manifest, or manifest with unfilled roles — both render as briefing alerts. No more T-2 catches by you.

Hard prerequisite

Conductor's fuzzy-fallback bug must be fixed first (or in the same PR)

scripts/conductor-agent.js:422-440 uses substring matching to dispatch actions — meaning a future crew_manifest_v2 message would silently route to the v1 handler. That's exactly the silent-failure class this proposal is trying to eliminate. The fix is mechanical (one file, ~10 lines), it's already on the NEXT-SESSION queue as item #10, and this proposal depends on it. If you approve the proposal, the fuzzy-fallback fix is implicit.

The 8 decisions

Q1
Normalized crew_assignments table, keyed on (project_id, shoot_date)
Defer extracting a proper shoots table until Phase 2 — minimum viable change first.
Recommended yes
Why not a JSON blob? The whole point of this structure is queryability — "what's unfilled in the next 7 days across all projects" is a one-line SQL query against a normalized table, but a blob-walk against every row of every project's JSON every morning at 5:30 AM.

Why not a proper shoots table first

Conductor today has no shoots table — shoots are JSON arrays inside retainer_cycles.shoot_dates. Extracting that into a first-class table is a substantial migration touching the entire briefing pipeline. Per Karpathy #2 (simplicity first), we ship crew_manifest with a tuple key now and defer the shoots-table extraction to Phase 2.

Tradeoff cost: if a shoot date is rescheduled, every crew_assignment row for the old date orphans. A migration script handles it with one UPDATE. Document the reschedule procedure.
Q2
Graceful-degrade if ACD can't extract crew from context
Strategy still emits; manifest is skipped with a warning; T-7 audit catches the missing manifest later.
Recommended yes

The scenario

Early-stage projects where talent isn't booked, location is TBD, or crew references just aren't in the payload context. ACD's composer can't fabricate a manifest from thin air.

The choice

Block strategy emission until crew can be solved? That breaks every new client where context is sparse. Better: emit strategy, return status: "insufficient_context" on the manifest side, let the T-7 audit catch the missing manifest as a "no manifest at all" alert (§5 of the proposal).

Coupling strategy to manifest at producer-side is too tight. Make absence observable, not blocking.
Q3
ACD sends crew_manifest directly to conductor (not via Kai)
Skip the Kai-as-relay hop; conductor owns the storage.
Recommended yes

Why direct routing

  • Conductor owns the DB. Kai-as-relay means two hops where one suffices.
  • Kai's briefing already reads from conductor DB. Adding a forwarding path is dead weight.
  • Matches the existing file-drop pattern direction (~/.kameha/shared/acd-production-briefing.json is also ACD → consumer).

Mesh route delta

Add to scripts/mesh/mesh-api.js seedRoutePermissions():

['acd', 'conductor', 1, 'crew_manifest deliverable. Tier 1 — informational, auto-deliver.']

Q4
Conductor's fuzzy-fallback fix is a hard prerequisite
Lands first or in the same PR. No exceptions.
Recommended yes

The bug, in plain English

Conductor's dispatch table uses action.includes(capability) — meaning the action name crew_manifest_v2 would substring-match crew_manifest and run the v1 handler. That's a silent corruption class — exactly what this proposal is built to eliminate.

If the fix doesn't land alongside crew_manifest: we deploy a contract that's structurally guaranteed to break the first time we version it. Self-defeating.

Cost of the fix

Single file. ~10 line change. Already triaged on NEXT-SESSION as item #10. Effectively free if bundled.

Q5
Delete-then-insert upsert (whole-document replace per emission)
When ACD re-emits a manifest, conductor deletes old rows for that shoot and inserts fresh.
Recommended yes

Why not row-level upsert

Row-level upsert invites ambiguity: if ACD's new manifest doesn't include a role that was in the old one, did ACD intentionally drop it or accidentally forget?

Whole-document replace is simpler: the latest manifest from ACD is the authoritative truth. Matches how ACD composes — it regenerates from scratch each time, not as a diff.

Implementation

DELETE FROM crew_assignments WHERE project_id=X AND shoot_date=Y; INSERT new rows; wrapped in a SQLite transaction.

Q6
Audit-time enforcement of the "no manifest" invariant
T-7 briefing alert flags confirmed shoots that have no manifest at all — not a hard block at confirm-time.
Recommended yes

Three options considered

  • Block at confirm-time: conductor refuses to confirm a shoot without a manifest. Hard guarantee, but breaks every existing shoot that has no manifest today.
  • Block at producer-time: ACD refuses to emit production_strategy without a manifest. False positives for sparse-context projects.
  • Audit-time (chosen): the morning briefing renders confirmed-shoot-no-manifest as a top-bucket alert. Observable, doesn't break flow.
This is the same pattern as the action-vocab design's silent-failure detection — make the absence observable, not the presence required.
Q7
Split the implementation work — CA opens PRs for library/tests/schema; you apply daemon-side edits
Owners.json policies dictate the split. No mechanical bypass on contract-touching files.
Recommended yes

Q8
5-phase rollout with a feature flag on ACD's emission
Each phase is independently reversible. See timeline below.
Recommended yes

The flag (ACD_EMIT_CREW_MANIFEST=true) means we can deploy all the plumbing, verify it inert, then flip the producer side on. If anything breaks downstream, flag-off is the rollback.

Q1 · Registry home repo
Kai's knowledge/manifests/crew-manifest.schema.json
Conductor (the validator) lives in Kai — schema reads from same repo at runtime. ACD vendors a copy for composing; CA's drift-audit subcommand catches divergence.
Q2 · Who can edit a receiver's accepts block
Receiver-agent's owners.json gates writes
Conductor's accepts block lives under Kai's scripts/mesh/** — human_review_required. CA always drafts; you always review contract changes. Senders' sends blocks can be more freely modified (asymmetry intentional — receivers are load-bearing).
Q3 · First adoption pair
Replace "ACD→Framer creative_brief" with "ACD→conductor crew_manifest"
ACD→Framer is no longer urgent (route exists, session 2 closure). Crew_manifest is greenfield, you personally care, validation loop is short. Second pair stays as nami↔framer schedule_post_response (P0-A in overnight audit).
Q4 · Strict vs warn validation
Split posture — conductor warns, ACD strict
Conductor (receiver) stores even partially-valid manifests with validation_warnings populated — receiver-side strictness blocks data ingestion, too brittle. ACD (sender) refuses to emit malformed messages — sender-side strictness is cheap. Flip both to mutual strict after 2 weeks clean.
Q5 · Versioning
Date-versioned manifest_version (e.g., 2026-05-20.1); conductor supports last 2 versions
Envelope carries manifest_version. Conductor handles current + prior version concurrently. ACD emits latest only. Hard cutoff after 1 year — conductor drops prior-version branch.
